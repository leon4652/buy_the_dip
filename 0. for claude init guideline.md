## 3. 최우선 결정사항

다음 결정사항은 반드시 따른다.

1. Backend는 Spring Boot로 구현한다.
2. Frontend는 React와 Vite로 구현한다.
3. Backend와 Frontend는 REST API로 통신한다.
4. Database는 PostgreSQL 계열을 사용한다.
5. 눌림점수 계산은 반드시 Backend에서 수행한다.
6. Frontend에는 눌림점수 계산 로직을 두지 않는다.
7. Next.js 단일 앱 구조는 사용하지 않는다.
8. FastAPI 기반 Backend는 사용하지 않는다.
9. 초기 MVP에서는 로그인, 자동 매매, 실시간 시세, AI 추천, 뉴스 분석을 제외한다.
10. 초기 구현은 외부 API보다 Mock 데이터와 계산 로직 검증을 우선한다.

## 4. 프로젝트 목표

buy_the_dip은 개인용 주식 눌림 감시 서비스이다.

사용자는 관심 종목을 등록한다.

서비스는 각 종목의 일봉 데이터를 기반으로 기술 지표와 눌림점수를 계산한다.

사용자는 대시보드에서 관심 종목의 눌림점수, 상태, 핵심 사유를 확인한다.

눌림점수는 매수 추천 점수가 아니다.

눌림점수는 관심 종목이 상승 추세 안에서 조정 구간에 들어왔는지 판단하기 위한 보조 지표이다.

## 8. Backend 계층 구조

Backend는 단순한 계층형 구조로 구현한다.

Controller
  Service
    Domain Logic
      Repository
        Database

계층별 규칙은 다음과 같다.

| 계층 | 규칙 |
| --- | --- |
| Controller | HTTP 요청과 응답만 처리한다 |
| Service | 유스케이스와 트랜잭션을 처리한다 |
| Domain Logic | 계산 로직과 상태 분류를 처리한다 |
| Repository | DB 접근만 처리한다 |
| DTO | API 요청과 응답에 사용한다 |
| Entity | DB 매핑에 사용한다 |

Controller에 비즈니스 로직을 넣지 않는다.

Entity를 그대로 API 응답으로 반환하지 않는다.

계산 로직은 테스트 가능하게 분리한다.

## 21. 상태 분류 기준

상태는 다음 중 하나로 분류한다.

| 상태 | 의미 |
| --- | --- |
| 관찰 전 | 아직 눌림 조건에 충분히 들어오지 않은 상태 |
| 눌림 진입 | 상승 추세 안에서 가격 조정이 시작된 상태 |
| 매수 검토 | 눌림 깊이와 추세 조건이 비교적 양호한 상태 |
| 반등 확인 필요 | 가격 위치는 괜찮으나 재상승 신호가 부족한 상태 |
| 추세 훼손 | 단순 눌림이 아니라 하락 전환 가능성이 높은 상태 |

기본 점수 기준은 다음과 같다.

| 점수 | 상태 |
| --- | --- |
| 80 이상 | 매수 검토 |
| 65 이상 80 미만 | 눌림 진입 |
| 50 이상 65 미만 | 반등 확인 필요 |
| 35 이상 50 미만 | 관찰 전 |
| 35 미만 | 관찰 전 |

35점 미만이라도 강제 추세 훼손 조건이 없으면 추세 훼손으로 분류하지 않는다.

총점 80점 이상이라도 반등 확인 점수가 5점 미만이면 매수 검토가 아닌 반등 확인 필요로 분류한다.

강제 추세 훼손 조건은 다음과 같다.

1. 현재가가 60일 이동평균선보다 8퍼센트 이상 아래에 있음
2. 최근 60거래일 고점 대비 하락률이 25퍼센트를 초과함
3. 최근 5거래일 중 2일 이상 거래량 급증을 동반한 장대음봉 발생
4. 20일 이동평균선이 60일 이동평균선 아래로 내려감

## 24. 1차 구현 목표

1차 목표는 전체 서비스를 완성하는 것이 아니다.

1차 목표는 Backend와 Frontend가 분리된 상태에서 다음 흐름이 동작하는 것이다.

1. Backend가 실행된다.
2. Frontend가 실행된다.
3. Frontend가 Backend health API를 호출한다.
4. Mock 관심 종목 목록이 표시된다.
5. Mock 일봉 데이터로 눌림점수가 계산된다.
6. 대시보드에 종목명, 점수, 상태, 요약 문구가 표시된다.

DB 연결과 외부 API 연동은 이 흐름이 동작한 뒤 진행한다.

## 25. Mock 데이터 우선 원칙

초기에는 외부 주식 API를 바로 연결하지 않는다.

이유는 다음과 같다.

1. API 인증 문제로 개발 흐름이 깨질 수 있다.
2. 무료 한도 제한이 있을 수 있다.
3. 응답 포맷이 바뀔 수 있다.
4. 데이터 품질 문제로 계산 로직 검증이 어려울 수 있다.

따라서 초기에는 Backend 내부 Mock 데이터로 계산 로직을 먼저 검증한다.

## 26. 외부 데이터 API 원칙

Spring Boot Backend에서 직접 호출 가능한 HTTP API를 우선 검토한다.

후보는 다음과 같다.

| 후보 | 비고 |
| --- | --- |
| Alpha Vantage | HTTP API 제공, 무료 한도 제한 |
| Twelve Data | HTTP API 제공, 무료 한도 제한 |
| Polygon.io | 데이터 품질 우수, 비용 발생 가능 |
| Yahoo Finance 비공식 엔드포인트 | 구현 가능하나 안정성 주의 필요 |

yfinance와 FinanceDataReader는 Python 기반이므로 Spring Boot 핵심 의존성으로 두지 않는다.

## 31. AI 작업 시작 프롬프트

Claude Code 또는 Antigravity에 작업을 맡길 때 아래 내용을 시작 프롬프트로 사용한다.

프로젝트 루트는 C:\project\buy_the_dip 이다.

현재 레포지토리에는 docs 문서만 있고 소스 코드는 없다.

docs 문서를 읽고 Spring Boot Backend와 React Vite Frontend를 분리한 구조로 프로젝트를 생성하라.

Next.js 단일 앱 구조는 사용하지 않는다.

Backend는 Spring Boot, Java, Gradle, Spring Web, Spring Data JPA, PostgreSQL Driver 기반으로 생성한다.

Frontend는 React, TypeScript, Vite 기반으로 생성한다.

초기에는 DB와 외부 주식 API를 연결하지 말고 Mock 데이터로 동작하는 흐름을 먼저 만든다.

우선 구현 목표는 Backend health API, Frontend health API 호출, Mock 관심 종목 대시보드, Mock 일봉 데이터 기반 눌림점수 계산이다.

눌림점수 계산 로직은 반드시 Backend에 둔다.

Frontend는 API 응답을 표시만 한다.

구현 후 README.md에 실행 방법을 정리하라.

## 32. 1차 완료 기준
