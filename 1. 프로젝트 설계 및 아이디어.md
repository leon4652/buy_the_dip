# 프로젝트 설계 및 아이디어 v2

## 1. 문서 목적

이 문서는 buy_the_dip 프로젝트를 Claude Code 또는 Antigravity 같은 AI 개발 도구가 일관되게 이해하고 구현할 수 있도록 작성한 setup용 설계 문서이다.

이 문서는 기획 아이디어를 넓게 나열하는 문서가 아니다.

목적은 다음과 같다.

1. 프로젝트 목표를 명확히 고정한다.
2. MVP 범위를 명확히 제한한다.
3. 기술 스택 선택지를 제거하고 확정된 구조를 제시한다.
4. Backend와 Frontend의 책임을 분리한다.
5. 눌림점수 계산 기준을 구현 가능한 수준으로 정리한다.
6. AI 개발 도구가 임의로 Next.js 단일 앱, FastAPI 메인 서버, Python 중심 구조로 변경하지 않도록 한다.

## 2. 프로젝트 개요

프로젝트명은 buy_the_dip이다.

서비스 표시명은 바이더딥으로 한다.

바이더딥은 개인용 주식 눌림 감시 대시보드이다.

사용자는 관심 종목을 등록하고, 서비스는 해당 종목이 상승 추세 안에서 매수 검토 가능한 조정 구간에 들어왔는지 점수와 상태로 보여준다.

이 서비스는 매수 추천 서비스가 아니다.

이 서비스는 자동매매 서비스가 아니다.

이 서비스의 핵심 목적은 사용자가 차트를 매번 직접 확인하지 않아도 관심 종목이 현재 어느 상태에 있는지 빠르게 판단하도록 돕는 것이다.

## 3. 확정 결정사항

초기 MVP 기준 확정사항은 다음과 같다.

1. Backend는 Java와 Spring Boot로 구현한다.
2. Frontend는 React, TypeScript, Vite로 구현한다.
3. Backend와 Frontend는 분리한다.
4. Backend와 Frontend는 REST API로 통신한다.
5. Database는 PostgreSQL 계열을 사용한다.
6. 눌림점수 계산 로직은 Backend에 둔다.
7. Frontend에는 눌림점수 계산 로직을 두지 않는다.
8. 초기 시장은 미국 주식과 ETF를 우선 대상으로 한다.
9. 초기 가격 데이터는 최신 종가 기준 일봉 데이터를 사용한다.
10. 실시간 시세는 MVP에서 제외한다.
11. 로그인은 MVP에서 제외한다.
12. 자동매매는 MVP에서 제외한다.
13. 뉴스 요약은 MVP에서 제외한다.
14. AI 종목 추천은 MVP에서 제외한다.
15. 알림 기능은 MVP에서 제외한다.
16. 백테스트는 MVP에서 제외한다.
17. Next.js 단일 앱 구조는 사용하지 않는다.
18. FastAPI를 메인 Backend로 사용하지 않는다.
19. Python 중심 MVP 구조로 구현하지 않는다.

## 10. 핵심 개념

## 10.1 눌림

눌림은 상승 추세가 유지되는 종목이 단기적으로 가격 조정을 받는 상태를 의미한다.

이 서비스에서 눌림은 단순히 가격이 많이 하락한 상태를 의미하지 않는다.

눌림 판단에는 다음 요소가 함께 필요하다.

1. 상승 추세 유지
2. 적정 조정 깊이
3. 이동평균선 근접
4. 거래량 안정
5. 반등 확인 가능성

## 10.2 눌림점수

눌림점수는 관심 종목이 현재 매수 검토 가능한 눌림 구간에 있는지를 100점 만점으로 표현한 점수이다.

눌림점수는 매수 추천 점수가 아니다.

눌림점수는 관심 종목을 우선순위로 정렬하기 위한 필터링 점수이다.

## 10.3 상태

종목 상태는 다음 중 하나로 분류한다.

```text
관찰 전
눌림 진입
매수 검토
반등 확인 필요
추세 훼손
데이터 부족
```

## 11. 눌림점수 구조

눌림점수는 100점 만점이다.

```text
눌림점수 = 추세 유지 점수
         + 조정 깊이 점수
         + 이동평균선 근접도 점수
         + 거래량 안정도 점수
         + 반등 확인 점수
```

배점은 다음과 같다.

```text
추세 유지: 35점
조정 깊이: 25점
이동평균선 근접도: 15점
거래량 안정도: 15점
반등 확인: 10점
총점: 100점
```

## 12. 세부 점수 기준

## 12.1 추세 유지 점수

```text
현재 종가가 20일 이동평균선 위: 10점
20일 이동평균선이 60일 이동평균선 위: 10점
60일 이동평균선 기울기가 양수: 10점
최근 저점이 이전 저점보다 높음: 5점
최대 점수: 35점
```

## 12.2 조정 깊이 점수

기준 고점은 최근 60거래일 고점으로 한다.

고점 대비 하락률은 양수로 계산한다.

```text
0퍼센트 이상 3퍼센트 미만: 5점
3퍼센트 이상 8퍼센트 이하: 25점
8퍼센트 초과 15퍼센트 이하: 18점
15퍼센트 초과 25퍼센트 이하: 8점
25퍼센트 초과: 0점
```

## 12.3 이동평균선 근접도 점수

```text
현재 종가가 20일선과 3퍼센트 이내: 8점
현재 종가가 60일선과 5퍼센트 이내: 7점
최대 점수: 15점
```

## 12.4 거래량 안정도 점수

기준 거래량은 최근 20거래일 평균 거래량으로 한다.

```text
하락일 평균 거래량이 최근 20일 평균 거래량보다 낮음: 8점
최근 3거래일 동안 급락 거래량이 없음: 4점
반등일 거래량이 전일보다 증가: 3점
최대 점수: 15점
```

거래량 급증 기준은 최근 20거래일 평균 거래량의 1.8배 이상으로 한다.

## 12.5 반등 확인 점수

```text
종가가 전일 고가 돌파: 4점
종가가 5일 이동평균선 회복: 3점
RSI가 전일 대비 상승: 2점
양봉 마감: 1점
최대 점수: 10점
```

## 13. 상태 분류 규칙

상태 분류는 점수만으로 판단하지 않는다.

상태 분류는 다음 순서로 판단한다.

1. 데이터 부족 여부 판단
2. 강제 추세 훼손 조건 판단
3. 반등 확인 부족 여부 판단
4. 점수 구간 기준 상태 판단

## 13.1 데이터 부족

일봉 데이터가 60거래일 미만이면 데이터 부족으로 분류한다.

## 13.2 강제 추세 훼손 조건

다음 조건 중 하나라도 충족하면 총점과 무관하게 추세 훼손으로 분류한다.

1. 현재 종가가 60일 이동평균선보다 8퍼센트 이상 아래에 있음
2. 최근 60거래일 고점 대비 하락률이 25퍼센트를 초과함
3. 최근 5거래일 중 2일 이상 거래량 급증을 동반한 장대음봉 발생
4. 20일 이동평균선이 60일 이동평균선 아래로 내려감
5. 직전 스윙 저점을 종가 기준으로 이탈함

## 13.3 반등 확인 부족

총점이 높더라도 반등 확인 점수가 낮으면 매수 검토로 분류하지 않는다.

기준은 다음과 같다.

```text
총점 80점 이상이고 반등 확인 점수 5점 이상: 매수 검토
총점 80점 이상이고 반등 확인 점수 4점 이하: 반등 확인 필요
```

## 13.4 기본 점수 구간

강제 추세 훼손 조건이 없고 데이터가 충분할 경우 다음 기준을 적용한다.

```text
80점 이상: 매수 검토
65점 이상 80점 미만: 눌림 진입
50점 이상 65점 미만: 반등 확인 필요
35점 이상 50점 미만: 관찰 전
35점 미만: 관찰 전
```

35점 미만이라도 강제 추세 훼손 조건이 없으면 무조건 추세 훼손으로 보지 않는다.

## 14. 계산 로직 의사코드

## 14.1 눌림점수 계산

```text
function calculatePullbackScore(priceData):
    if priceData.length < 60:
        return status = DATA_INSUFFICIENT

    ma5 = movingAverage(close, 5)
    ma20 = movingAverage(close, 20)
    ma60 = movingAverage(close, 60)
    rsi14 = calculateRsi(close, 14)

    recentHigh60 = max(high over last 60 trading days)

    drawdown = (recentHigh60 - currentClose) / recentHigh60 * 100

    trendScore = calculateTrendScore(currentClose, ma20, ma60, priceData)
    drawdownScore = calculateDrawdownScore(drawdown)
    maDistanceScore = calculateMaDistanceScore(currentClose, ma20, ma60)
    volumeScore = calculateVolumeScore(priceData)
    reboundScore = calculateReboundScore(priceData, ma5, rsi14)

    totalScore = trendScore + drawdownScore + maDistanceScore + volumeScore + reboundScore

    if isTrendBroken(currentClose, ma20, ma60, drawdown, priceData):
        status = TREND_BROKEN
    else:
        status = classifyStatus(totalScore, reboundScore)

    summary = generateSummary(status, totalScore, drawdown, trendScore, reboundScore)

    return {
        totalScore,
        trendScore,
        drawdownScore,
        maDistanceScore,
        volumeScore,
        reboundScore,
        status,
        summary
    }
```

## 14.2 강제 추세 훼손 판단

```text
function isTrendBroken(currentClose, ma20, ma60, drawdown, priceData):
    if currentClose < ma60 * 0.92:
        return true

    if drawdown > 25:
        return true

    if ma20 < ma60:
        return true

    if hasMultipleHighVolumeBearishCandles(priceData, lastDays = 5):
        return true

    if closeBreaksPreviousSwingLow(priceData):
        return true

    return false
```

## 22. Claude 작업 지시용 요약

아래 내용은 Claude Code 또는 Antigravity 작업 시작 시 기준 지시로 사용할 수 있다.

```text
Spring Boot 백엔드와 React 프론트엔드를 분리한 개인용 주식 눌림 감시 웹 서비스를 만든다.

프로젝트명은 buy_the_dip이고 서비스 표시명은 바이더딥이다.

Backend는 Java 21, Spring Boot 3.x, Spring Web, Spring Data JPA, Gradle을 사용한다.

Frontend는 React, TypeScript, Vite, Tailwind CSS를 사용한다.

Database는 PostgreSQL 계열을 사용한다.

Backend는 관심 종목 관리, 종목 정보 관리, 일봉 데이터 저장, 기술 지표 계산, 눌림점수 계산, 상태 분류, 해석 문구 생성, REST API 제공을 담당한다.

Frontend는 Backend API 응답을 받아 관심 종목 대시보드와 종목 상세 화면을 표시한다.

눌림점수 계산 로직은 Frontend에 두지 않는다.

초기 MVP에서는 로그인, 자동매매, 실시간 시세, 뉴스 요약, AI 종목 추천, 알림, 백테스트를 제외한다.

Next.js 단일 앱 구조는 사용하지 않는다.

FastAPI를 메인 Backend로 사용하지 않는다.

Python 중심 MVP 구조로 구현하지 않는다.

초기 데이터는 샘플 일봉 데이터 또는 CSV로 계산 로직을 먼저 검증하고, 이후 외부 주식 데이터 API 연동을 검토한다.

drawdown은 최근 60거래일 고점 대비 하락률을 양수로 계산한다.

상태 분류는 데이터 부족, 강제 추세 훼손 조건, 반등 확인 부족, 점수 구간 순서로 판단한다.

직전 스윙 저점 이탈은 총점과 무관하게 추세 훼손 조건으로 본다.
```
