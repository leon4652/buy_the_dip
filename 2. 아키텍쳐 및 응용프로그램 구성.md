# 2. 아키텍쳐 및 응용프로그램 구성

## 1. 문서 목적

본 문서는 buy_the_dip 프로젝트의 아키텍쳐와 응용프로그램 구성을 정의한다.

이 문서는 Claude Code, Antigravity 등 AI 개발 도구가 프로젝트 구조, 기술 스택, 책임 범위, 구현 순서를 일관되게 이해하도록 하기 위한 기준 문서이다.

본 문서는 초기 MVP 기준으로 작성한다.

## 2. 프로젝트 개요

buy_the_dip은 개인용 주식 눌림 감시 서비스이다.

사용자는 관심 종목을 등록하고, 서비스는 해당 종목의 일봉 데이터를 기반으로 눌림점수와 상태를 계산한다.

눌림점수는 매수 추천 점수가 아니다.

눌림점수는 관심 종목이 상승 추세 안에서 조정 구간에 들어왔는지 판단하기 위한 보조 지표이다.

## 4. 제외한 구조

초기 검토 과정에서 Next.js 단일 앱 구조를 고려했으나 최종적으로 제외한다.

제외 사유는 다음과 같다.

1. 프로젝트 목적이 단순한 빠른 화면 제작이 아니다.
2. 개발자의 주력 스택인 Spring Boot 백엔드 역량을 살리는 것이 중요하다.
3. 눌림점수 계산, 데이터 수집, 캐싱, 배치 처리는 Backend 중심 구조가 더 적합하다.
4. 포트폴리오와 학습 자산 관점에서 Spring Boot 기반 구조가 더 의미 있다.

따라서 본 프로젝트는 Spring Boot Backend와 React Frontend를 분리한 구조로 진행한다.

## 7. Backend 구성

## 7.1 Backend 책임

Backend는 다음 책임을 가진다.

1. 관심 종목 관리
2. 종목 기본 정보 관리

4. 일봉 데이터 저장
5. 기술 지표 계산
6. 눌림점수 계산
7. 상태 분류
8. 해석 문구 생성
9. Frontend용 REST API 제공
10. 향후 배치 처리와 캐싱 확장

## 7.2 Backend 기술 스택

| 영역 | 기술 |
| --- | --- |
| Language | Java |
| Framework | Spring Boot |
| Web | Spring Web |
| Persistence | Spring Data JPA |
| Database Driver | PostgreSQL Driver |
| Build Tool | Gradle |
| Test | JUnit, Spring Boot Test |

초기에는 다음 기술을 사용하지 않는다.

1. Spring Security

## 7.3 Backend 계층 구조

Backend는 단순한 계층형 구조를 따른다.

```text

  ↓
Service
  ↓
Domain Logic
  ↓
Repository
  ↓
Database
```

각 계층의 책임은 다음과 같다.

| 계층 | 책임 |
| --- | --- |
| Controller | HTTP 요청과 응답 처리 |
| Service | 유스케이스 처리와 트랜잭션 관리 |
| Domain Logic | 눌림점수 계산, 상태 분류, 지표 계산 |

## 9. Database 구성

## 9.1 Database 선택

Database는 PostgreSQL 계열을 사용한다.

초기 후보는 다음과 같다.

1. Local PostgreSQL
2. Supabase PostgreSQL

권장 방향은 Supabase PostgreSQL이다.

이유는 다음과 같다.

1. 별도 DB 서버 관리 부담이 적다.
2. 추후 배포와 연동이 쉽다.
3. Backend에서는 일반 PostgreSQL처럼 접근할 수 있다.

단, Backend는 Supabase 고유 기능에 강하게 의존하지 않는다.

## 10. 데이터 수집 구성

## 10.1 원칙

외부 주식 데이터는 Backend에서 수집한다.

Frontend는 외부 주식 데이터 API를 직접 호출하지 않는다.

## 10.2 초기 데이터 소스 후보

Spring Boot Backend에서 사용 가능한 HTTP API 계열을 우선 검토한다.

후보는 다음과 같다.

| 후보 | 비고 |
| --- | --- |
| Alpha Vantage | HTTP API 제공, 무료 한도 제한 |
| Twelve Data | HTTP API 제공, 무료 한도 제한 |
| Polygon.io | 데이터 품질 우수, 비용 발생 가능 |
| Yahoo Finance 비공식 엔드포인트 | 구현은 가능하나 안정성 주의 필요 |
| FinanceDataReader | Python 기반 실험 후보 |
| yfinance | Python 기반 실험 후보 |

Spring Boot 중심 구조에서는 yfinance를 직접 핵심 의존성으로 두지 않는다.

yfinance는 Python 실험 스크립트 또는 검증용 도구로만 고려한다.

## 10.3 초기 권장 방향

초기 구현에서는 다음 순서를 따른다.

1. Backend 내부에 Mock Stock Data Provider 구현
2. 고정 샘플 데이터로 눌림점수 계산 검증
3. HTTP 기반 외부 데이터 API 연동
4. DB 저장과 캐싱 추가
5. 자동 갱신 스케줄러 추가

이 순서가 가장 안전하다.

외부 API 연동부터 시작하면 API 한도, 인증, 데이터 형식 문제로 MVP 속도가 느려질 수 있다.

## 12. API 구성

## 12.1 초기 API

초기 API는 다음을 기준으로 한다.

| Method | Path | 설명 |
| --- | --- | --- |
| GET | /api/health | Backend 상태 확인 |
| GET | /api/watchlist | 관심 종목 목록 조회 |
| POST | /api/watchlist | 관심 종목 추가 |
| DELETE | /api/watchlist/{watchlistId} | 관심 종목 삭제 |
| GET | /api/stocks/{stockId} | 종목 상세 조회 |
| GET | /api/stocks/{stockId}/prices | 종목 일봉 데이터 조회 |
| GET | /api/stocks/{stockId}/scores/latest | 최신 눌림점수 조회 |
| POST | /api/stocks/{stockId}/refresh | 종목 데이터 수동 갱신 |
| POST | /api/scores/refresh | 전체 관심 종목 점수 수동 갱신 |

API 식별자는 symbol이 아닌 id(stockId, watchlistId) 기반을 사용한다.

## 12.2 API 설계 원칙

1. Frontend는 Backend API만 호출한다.
2. Backend 응답은 화면 표시 기준으로 필요한 값을 포함한다.
3. 계산 근거를 함께 제공하여 사용자가 점수 이유를 알 수 있게 한다.
4. 초기에는 인증을 제외한다.
5. 추후 사용자 계정이 추가되면 watchlist를 사용자별로 분리한다.

## 17. AI 개발 도구 작업 기준

Claude Code와 Antigravity는 다음 기준을 따라 작업한다.

1. 먼저 docs 문서를 읽고 작업한다.
2. Backend 계산 로직을 Frontend로 옮기지 않는다.
3. Controller에 비즈니스 로직을 넣지 않는다.
4. Service와 Domain Logic을 분리한다.
5. 외부 API 호출 코드는 client 패키지에 둔다.
6. API 응답 DTO와 Entity를 분리한다.
7. 초기에는 과도한 추상화를 만들지 않는다.
8. 작게 동작하는 기능 단위로 구현한다.
9. 테스트 가능한 계산 로직부터 구현한다.

## 20. 최종 원칙

본 프로젝트의 중심은 Backend이다.

React Frontend는 화면 표현을 담당한다.

Spring Boot Backend는 데이터 수집, 계산, 상태 분류, API 제공을 담당한다.

초기 목표는 완성도 높은 대규모 서비스를 만드는 것이 아니다.

초기 목표는 Spring Boot 기반으로 동작하는 개인용 눌림 감시 MVP를 빠르게 구현하는 것이다.
